# Part A ‚Äî The live site (`msail.github.io`): what a redesign actually replaces

## A1. Stack, build & deploy ‚Äî the real pipeline (and its bus factor)

- **Generator:** Hugo with the **Academic** theme v4.8.0 (`<meta name="generator" content="Source Themes Academic 4.8.0">`). "Academic" later became **Wowchemy**, then **HugoBlox** ‚Äî and the original Academic theme is **end-of-life / unmaintained**. So the live site is on a frozen, deprecated theme generation (circa 2020‚Äì2021). Assets: `/css/academic.css`, `/css/all.min.css` (Font Awesome), Bootstrap, a client-side `index.json` search index.
- **Source of truth:** **`Nivedhitha-dp/MSAIL`** ‚Äî a **personal GitHub account**, not the MSAIL org. (Nivedhitha is listed on the live About page as "Mentorship & Web Experience Lead" and is the author of the most recent publish commits.) An older `ForkBomb20/MSAIL` (2024) looks like a prior maintainer's copy. **The org does not own its own website source.** This is the single biggest operational risk (see A4).
- **Publish mechanism:** maintainers run Hugo **locally**, then **hand-commit the built `public/` output** into `MSAIL/MSAIL.github.io` on branch `master`. That repo's description literally warns "DO NOT MODIFY: Autogenerated content, may be overwritten without warning." There is **no GitHub Actions / CI** (`.github/workflows` ‚Üí 404; Pages `build_type=legacy`, `source=master/` root). Every deploy is a manual local build + manual commit.
- **Serving:** classic GitHub Pages off `MSAIL/MSAIL.github.io@master`. Pages API reports **`cname=null`** (no custom domain) and **`https_enforced=false`** ‚Äî the site is reachable over plain **HTTP**, and the canonical `SITEURL`/sitemap base is `http://msail.github.io/`. Enabling "Enforce HTTPS" is a one-click, no-excuse fix.
- **Continuity from 2016:** the publish *target* (`MSAIL/MSAIL.github.io`) is the same org Pages repo the old Pelican site used; only the upstream source changed (Pelican-in-org ‚Üí Hugo-in-personal-acct). The 2016 `-source` repo was simply orphaned.

## A3. Design & technical read (live) ‚Äî _medium confidence; network inspection only_

- **Responsive baseline exists** (unlike the 2016 repo): the live head has `<meta name="viewport" content="width=device-width, initial-scale=1">` and the Academic theme is Bootstrap-grid responsive. So mobile is *functional*, not broken.
- **Dated theme.** Academic 4.8.0 is a 2020‚Äì2021-era look (the familiar dark hero + centered card sections seen across thousands of academic/club sites). It reads "generic Hugo Academic template," not a distinctive MSAIL brand. The theme is also **unmaintained upstream**, so security/compat fixes have stopped.
- **Served over HTTP, not HTTPS** (Pages `https_enforced=false`) ‚Äî a flat-out defect in 2026; fix immediately regardless of redesign.
- **Stale chrome:** footer reads "**Copyright MSAIL 2023**" while content runs through Fall 2025 ‚Äî the boilerplate year is hardcoded and rotting.
- **Content freshness is uneven:** homepage advertises a "Fall 2025 Calendar," but the Talks archive stops at **March 2024**, the Resources class list is stamped "Last updated 9/23/21" with a "under construction" section, and the Andrew Ng quote from the 2016 site **survived the migration** onto the Talks page. The site looks maintained in places and abandoned in others.
- **Not assessed (no source / no audit run):** full CSS quality, JS weight, image optimization, color-contrast/WCAG conformance, Lighthouse performance. These should be measured directly against the Hugo source before redesign decisions; treat A3 as directional, not a verdict.

## A4. Live-site risks & landmines for a rebuild (ranked)

1. **The org doesn't control its own source (governance / bus factor) ‚Äî CRITICAL.** The Hugo source lives in `Nivedhitha-dp/MSAIL`, a personal account. When that student graduates or revokes access, the org can edit the *rendered* output by hand but effectively loses the ability to rebuild the site. **Before anything else: move the current Hugo source into the `MSAIL` org and confirm who has admin.** A redesign that can't locate/own the source is a non-starter.
2. **Manual build-and-commit publishing ‚Äî HIGH.** No CI means deploys depend on one person running Hugo locally with the right theme version and pushing built artifacts into a repo labeled "do not modify." It's fragile, unauditable, and unreproducible (no pinned Hugo/theme version visible). Replace with a GitHub Actions ‚Üí Pages workflow that builds from source on push.
3. **Deprecated theme ‚Äî HIGH.** Academic 4.8.0 is EOL; its successor lineage (Wowchemy ‚Üí HugoBlox) changed config formats. Upgrading in place is real work; this is a natural moment to choose the long-term stack deliberately rather than chase the theme's migrations.
4. **HTTP-only / no custom domain ‚Äî MEDIUM (but trivial to fix).** Turn on Enforce HTTPS now. If a custom domain (e.g. an `msail.org`-style domain ‚Äî note the 2016 repo's history shows a CNAME was once used and removed) is ever added, the CNAME must live in the published Pages repo and survive each rebuild.
5. **Don't nuke the Pages repo ‚Äî MEDIUM.** `MSAIL/MSAIL.github.io@master` carries the Pages serving config. Force-push a fresh tree if needed, but don't delete/recreate the repo (you'd lose Pages setup and any domain verification).
6. **URL/SEO continuity ‚Äî MEDIUM.** The live routes (`/talk/`, `/projects/`, `/aboutus/`, dated talk permalinks) are what Google and inbound links know. A new permalink scheme should ship redirects.

## B2. Stack, build pipeline & deploy (traced)

**SSG = Pelican.** `pelicanconf.py` is a Pelican settings module; `Makefile:1` runs `pelican`; `develop_server.sh`/`fabfile.py` import `pelican.server`. Build reads Markdown from `content/` (`PATH='content'`, `pelicanconf.py:17`), applies the local theme (`THEME="./theme"`), writes static HTML. `content/events/*.md` ‚Üí **articles** (`ARTICLE_PATHS=['events']`, line 28); `content/pages/*.md` ‚Üí **pages**. `LOAD_CONTENT_CACHE=False` forces full rebuilds.

**Two-repo submodule deploy (classic Pelican pattern).** Source lives here; rendered HTML goes to the org Pages repo `MSAIL/MSAIL.github.io`, wired as a git submodule (`.gitmodules`: `url=git@github.com:MSAIL/MSAIL.github.io.git`, **SSH**).

| Target | Action | Source |
|---|---|---|
| `make html` | `pelican content -o $(OUTPUTDIR) -s pelicanconf.py` (dev build into the submodule dir) | `Makefile:64-65` |
| `make publish` | same, `-s publishconf.py` (prod build into the submodule) | `Makefile:99-100` |
| `make github` | `publish`, then `cd MSAIL.github.io && git add -A && git commit -m "publish site" && git push origin master` | `Makefile:120-121,28` |

`make github` commits/pushes **inside the submodule's own working tree** ‚Äî the same manual "build then commit output" pattern the live Hugo site still uses today.

**Dead config ‚Äî `OUTPUT_PATH`.** `pelicanconf.py:48` sets `OUTPUT_PATH="/MSAIL.github.io"` (leading slash ‚Üí absolute path at disk root), but it's **never used**: every Makefile build passes `-o $(OUTPUTDIR)` (`$(BASEDIR)/MSAIL.github.io`), and Pelican's CLI `-o` overrides the config value. A bare `pelican -s pelicanconf.py` (no `-o`) would try to write to `/MSAIL.github.io` and fail on permissions. _(Verified.)_

**Dead tooling ‚Äî `fabfile.py`.** Stock `pelican-quickstart` Fabric boilerplate. **Cannot import on Python 3** (`fabfile.py:6 import SocketServer` is Py2-only; `python3 -c "import SocketServer"` ‚Üí `ModuleNotFoundError`), needs Fabric 1.x, and **contradicts the Makefile** (`deploy_path='output'` vs `MSAIL.github.io`; `gh-pages` + `ghp-import` vs `master` + in-submodule commit). Retains placeholder creds (`root@localhost`, `my_rackspace_username`). The `ssh/rsync/dropbox/ftp/s3/cf_upload` Makefile targets are likewise untouched boilerplate. _(Verified.)_

**Reproducibility gap.** No dependency manifest anywhere (no `requirements.txt`/`Pipfile`/`pyproject.toml`/`runtime.txt`/`.python-version`). Original Pelican/Jinja/Python versions unrecorded (circa 2016, likely Pelican 3.6.x). Nothing installed locally (`import pelican` ‚Üí `ModuleNotFoundError`); local Python is 3.14.5. A 2026 `pip install pelican` yields Pelican 4.x/5.x with breaking changes ‚Äî this tree is **not deterministically rebuildable**. _(Verified.)_

**VCS landmine ‚Äî orphan `output` gitlink.** `git submodule status` errors `fatal: no submodule mapping found in .gitmodules for path 'output'`: there are **two** mode-`160000` gitlinks (`MSAIL.github.io`, mapped; `output`, unmapped). ## Recommendation

**Rebuild ‚Äî but understand you are not "rebuilding a Pelican site," because you already left Pelican years ago.** This repo (Pelican, 2016) is a dead ancestor and should be archived, not revived: it is unreproducible (no pinned deps, original version unknown, nothing installed, Python now 3.14), VCS-broken (an orphan `output` gitlink that fails `git submodule`/recursive clone today), and its presentation layer fails every modern baseline at once (non-responsive, inaccessible, off-brand, a 521 KB avatar, HTTP, a deceptive build-time "last updated"). Staying on Pelican is moot; the real decision is "adopt-and-modernize the current Hugo site" vs "fresh rebuild," and given the EOL theme and the ownership/deploy problems, a deliberate fresh rebuild that first **rescues and re-homes the source** is the lower-risk long-term path.
